    50.043 - Vector Database¶

    Learning Outcomes¶

    By the end of this unit, you should be able to

    1. Explain the concept of embedding and why vector representation is useful.
    2. Describe the distance/similarity metrics used in vector search.
    3. Explain the limitations of traditional indexing (B+ Tree, Hash) for high-dimensional data.
    4. Describe the principles of approximate nearest neighbor (ANN) search.
    5. Compare different vector indexing structures: IVF, HNSW, and PQ.
    6. Understand the trade-offs between recall, latency, and memory in vector databases.

    Motivation¶

    Traditional relational databases and NoSQL systems excel at structured and semi-structured data. They rely on exact matching, range queries, and join operations. However, a growing class of applications requires a fundamentally different type of query: similarity search.

    Consider these use cases:

    • Semantic search: Find documents that mean the same thing, even if they share no keywords.
    • Image retrieval: Find images visually similar to a query image.
    • Recommendation: Find items similar to what a user previously liked.
    • AI and LLM applications: Retrieve contextually relevant passages for retrieval-augmented generation (RAG).

    In all these cases, the data and the query are represented as vectors (lists of real numbers in a high-dimensional space), and the goal is to find the data points closest to the query vector. This is the core problem that vector databases solve.

    From Data to Vectors: Embeddings¶

    What is an Embedding?¶

    An embedding is a dense vector representation of a data object, where semantically similar objects are mapped to nearby points in the vector space.

    1
2
3
    Document "machine learning tutorial"    ->  [0.12, -0.45, 0.83, ..., 0.21]   (d=768)
Document "AI education guide"          ->  [0.15, -0.42, 0.80, ..., 0.19]   (d=768)
Document "recipe for chocolate cake"   ->  [-0.61, 0.33, -0.12, ..., 0.77]  (d=768)

    The first two documents are close in the vector space because they are semantically related, while the third is far away.

    Note that all three documents produce vectors of the same dimensionality \(d = 768\). This is because an embedding model always outputs a fixed-size vector regardless of input length. A short sentence and a long article will both map to a 768-dimensional vector when passed through the same model. The dimensionality is determined by the model architecture.

    How Are Embeddings Created?¶

    Embeddings are produced by trained machine learning models:

    • Text: Sentence Transformers (e.g., Sentence-BERT), OpenAI Ada embedding, E5
    • Images: ResNet, CLIP visual encoder
    • Audio: Wav2Vec, Whisper embeddings
    • Tabular: Learned representations from neural networks

    The dimensionality \(d\) of the embedding vector is a property of the model, commonly ranging from 128 to 4096. Once generated, the embeddings are stored in the vector database as fixed-length arrays of floating-point numbers.

    Similarity and Distance Metrics¶

    To find "nearest" vectors, we need a metric to measure closeness. The three most common are:

    Euclidean Distance (L2)¶

    The straight-line distance between two vectors:

    \[ d_{L2}(\vec{a}, \vec{b}) = \|\vec{a} - \vec{b}\|_2 = \sqrt{\sum_{i=1}^{d} (a_i - b_i)^2} \]

    Smaller distance means more similar. This is the default metric in many systems.

    Cosine Similarity¶

    Measures the angle between two vectors, ignoring their magnitude:

    \[ \text{cosine}(\vec{a}, \vec{b}) = \frac{\vec{a} \cdot \vec{b}}{\|\vec{a}\|_2 \cdot \|\vec{b}\|_2} = \frac{\sum_{i=1}^{d} a_i b_i}{\sqrt{\sum_{i=1}^{d} a_i^2} \cdot \sqrt{\sum_{i=1}^{d} b_i^2}} \]

    Value ranges from \(-1\) (opposite) to \(1\) (identical direction). Often converted to a distance by \(1 - \text{cosine}(\vec{a}, \vec{b})\).

    Dot Product (Inner Product)¶

    \[ \text{dot}(\vec{a}, \vec{b}) = \vec{a} \cdot \vec{b} = \sum_{i=1}^{d} a_i b_i \]

    Larger value means more similar. Equivalent to cosine similarity when vectors are L2-normalized.

    Choosing a Metric¶

    Metric Use Case Notes
    L2 Geometric proximity Sensitive to vector magnitude
    Cosine Directional similarity Magnitude-invariant
    Dot Product LLM embeddings Requires normalized vectors to equal cosine

    The Curse of Dimensionality¶

    Before discussing indexing, we must understand why traditional index structures break down for high-dimensional vectors.

    Why B+ Trees Fail¶

    Recall from the access method unit, B+ Trees support efficient point queries and range queries. For a 1D or 2D key space, a B+ Tree can narrow down search by repeatedly splitting the search space.

    However, in high-dimensional space (\(d \geq 100\)), the volume of the space grows exponentially. Most points become nearly equidistant from each other. A range filter on one dimension eliminates very few candidates, and combining \(d\) such filters is computationally expensive.

    Why Hash Tables Fail¶

    Hash tables support exact matching. Two vectors that are "similar" (close in distance) will hash to completely unrelated buckets. Hash-based approaches only work with specialized techniques like locality-sensitive hashing (LSH), which is less commonly used in practice today.

    The Brute-Force Baseline¶

    The simplest approach is linear scan: compute the distance between the query and every stored vector, then return the \(k\) smallest.

    \[ \text{Cost} = N \times d \times O(1) \]

    where \(N\) is the number of vectors and \(d\) is the dimensionality.

    For \(N = 10^7\) and \(d = 768\), this requires approximately \(7.68 \times 10^9\) floating-point operations per query. While modern CPUs with SIMD can process this in milliseconds, scaling to billions of vectors makes linear scan impractical.

    Approximate Nearest Neighbor (ANN)¶

    To achieve fast search at scale, vector databases sacrifice exactness for speed. ANN algorithms return results that are approximately the nearest neighbors, with configurable recall.

    Recall vs. Latency Trade-off¶

    \[ \text{Recall} = \frac{|\text{ANN results} \cap \text{Ground truth top-k}|}{k} \]

    Higher recall requires examining more vectors, increasing latency. The key engineering challenge is to achieve high recall (e.g., 95%+) with sub-millisecond latency on millions of vectors.

    Vector Indexing Structures¶

    Three families of indexing structures dominate modern vector databases.

    Inverted File Index (IVF)¶

    Core Idea¶

    IVF adapts the concept of clustering from classical machine learning. The vector space is partitioned into \(n\) clusters using k-means. Each cluster has a centroid, and every vector is assigned to the nearest centroid.

    Query processing: 1. Compute which \(n\_probe\) centroids are closest to the query. 2. Search only the vectors in those \(n\_probe\) clusters. 3. Return the top-k from the combined candidates.

    graph TD
    Q["Query vector q"] --> C["Find nearest n_probe centroids"]
    C --> P1["Probe cluster 1"]
    C --> P2["Probe cluster 2"]
    C --> P3["Probe cluster n_probe"]
    P1 --> M["Merge candidates"]
    P2 --> M
    P3 --> M
    M --> R["Return top-k nearest"]

    Parameters¶

    • Number of clusters (\(n\)): More clusters means smaller lists and faster search, but requires building centroids via k-means. A rule of thumb is \(n \approx N/256\).
    • n_probe: Number of clusters to probe. Higher \(n\_probe\) improves recall but increases latency. \(n\_probe = 1\) means searching only one cluster; \(n\_probe = n\) means exhaustive search.

    Advantages and Limitations¶

    Pros Cons
    Simple to implement Requires k-means training on data
    Low memory overhead Performance degrades with uneven clusters
    Good for moderate-dimensional data Struggles with very high dimensions (\(d > 200\))

    Hierarchical Navigable Small World (HNSW)¶

    Core Idea¶

    HNSW is based on the small-world network theory, extended into a multi-layer graph structure.

    The index is a layered graph: * Layer 0 (bottom): Contains all vectors, densely connected. * Higher layers: Contain subsets of vectors, sparsely connected, acting as express highways. * A vector appears in at most one layer above 0, with probability decreasing exponentially with layer height.

    Query processing: 1. Start at the highest layer, find the entry point. 2. Greedily move to the neighbor closest to the query. 3. When no closer neighbor exists, drop to the next lower layer. 4. Repeat until layer 0, then return top-k from the local neighborhood.

    graph TD
    classDef layer fill:#e8f4f8,stroke:#333,stroke-width:1px;
    L3["Layer 3:   A --------- B"]:::layer
    L2["Layer 2: A -- C -- D -- B"]:::layer
    L1["Layer 1: A - C - E - D - F - B"]:::layer
    L0["Layer 0: A-B-C-D-E-F-G-H-I-J (dense graph)"]:::layer
    Q["Query q enters at Layer 3"] --> L3
    L3 --> L2
    L2 --> L1
    L1 --> L0
    L0 --> R["Return top-k"]

    Parameters¶

    • M: Maximum number of connections per node. Controls graph density. Typical: 16.
    • ef_construction: Neighborhood size during index building. Higher values create better graphs but slow construction. Typical: 200.
    • ef_search: Neighborhood size during query. Higher values improve recall but increase latency. Typical: 64.

    Advantages and Limitations¶

    Pros Cons
    State-of-the-art recall/latency ratio Higher memory overhead (graph edges)
    No training required (online insertion) Write-heavy workloads can be slow
    Supports dynamic insertion Tuning M and ef_search requires experimentation

    Product Quantization (PQ)¶

    Core Idea¶

    PQ is a compression technique that reduces the memory footprint of each vector, enabling more vectors to fit in RAM and faster distance computation.

    The \(d\)-dimensional vector is divided into \(m\) sub-vectors of dimension \(d/m\). Each sub-vector is independently quantized to one of \(256 = 2^8\) cluster centroids, using k-means. The vector is represented by \(m\) bytes (one byte per sub-vector index).

    \[ \text{Compression ratio} = \frac{d \times 4 \text{ bytes}}{m \text{ bytes}} \]

    For \(d = 768\) and \(m = 96\), we compress from 3072 bytes (float32) to 96 bytes -- a 32x reduction.

    Distance computation uses a precomputed lookup table. For each of the \(m\) sub-vector dimensions, we precompute the distance between all 256 query centroids and all 256 database centroids, stored in a \(256 \times 256\) table. Distance estimation is then \(m\) table lookups instead of \(d\) floating-point subtractions and multiplications.

    1
2
3
4
5
    Original: [a1, a2, ..., a768]       (3072 bytes, float32)
          |  sub1  | sub2 | ... | sub96 |

Quantized: [c1, c2, ..., c96]       (96 bytes, uint8)
           where ci in {0, 1, ..., 255}

    Advantages and Limitations¶

    Pros Cons
    Massive memory savings (8-32x) Approximation error in distance
    Faster distance computation Requires k-means training
    Often combined with IVF or HNSW Quality depends on sub-vector alignment

    A modern vector database is not just an index structure. It is a full data management system with these layers:

    Storage Layer¶

    Vectors are stored as binary arrays. Many systems support: * Flat storage: Linear scan, no index. Useful for small datasets or as a baseline. * Disk-based index: For datasets exceeding RAM, indices are persisted to disk (e.g., DiskANN). * Hybrid storage: Metadata (IDs, filters) in a relational store, vectors in a columnar format.

    Index Layer¶

    One or more index structures (IVF, HNSW, PQ, or combinations) are maintained per collection. Indexes are built incrementally or in batch.

    Query Layer¶

    Supports: * Nearest neighbor search: SELECT * FROM vectors ORDER BY vector <-> query LIMIT k * Scalar filtering: WHERE category = 'electronics' AND price > 100 * Hybrid search: Combine keyword search (BM25) with vector search, reranked by a fusion function.

    Concurrency and Consistency¶

    Vector databases face the same challenges as traditional databases: * Read concurrency: Multiple queries can scan the index simultaneously (read-only index, no locking needed). * Write handling: New vectors are often buffered and periodically merged into the index (similar to LSM-tree design). * Consistency: Most vector databases offer eventual consistency. Exact strong consistency is rare due to the approximate nature of ANN.

    Comparison of Index Structures¶

    Structure Best For Recall Latency Memory Write Support
    IVF Moderate dimensions, static data Medium-High Fast Low Batch
    HNSW High recall requirements Very High Medium High Online
    PQ Memory-constrained Medium Fastest Lowest Offline
    IVF+PQ Large-scale, cost-sensitive Medium Fast Low Batch
    HNSW+PQ Balanced High Medium Medium Online

    Practical Considerations¶

    Choosing an Index¶

    • Dataset size < 100K vectors: Flat index (linear scan) may be sufficient.
    • 100K - 10M vectors: IVF or HNSW.
    • > 10M vectors: HNSW or IVF+PQ.
    • Memory is constrained: Use PQ compression.

    Tuning for Performance¶

    1. Normalize vectors if using dot product or cosine similarity.
    2. Choose n_probe (IVF) or ef_search (HNSW) based on the recall/latency budget.
    3. Batch index building for better quality (especially IVF, which benefits from well-converged k-means).
    4. Profile with representative queries -- recall varies across the dataset.

    Integration with LLM Workflows¶

    Vector databases are a core component of Retrieval-Augmented Generation (RAG):

    graph LR
    D["Document corpus"] --> E["Embedding model"]
    E --> V["Vector database"]
    U["User question"] --> E2["Embedding model"]
    E2 --> Q["Query vector"]
    Q --> V
    V --> R["Top-k relevant passages"]
    R --> L["LLM with context"]
    L --> A["Answer"]

    The vector database retrieves relevant context, which the LLM uses to generate grounded, factually accurate responses. This reduces hallucination and enables domain-specific knowledge without fine-tuning.

    Summary¶

    Vector databases address a fundamentally different query paradigm from relational and NoSQL databases: approximate similarity search in high-dimensional space. Key takeaways:

    • Embeddings transform unstructured data into fixed-length vectors where proximity reflects semantic similarity.
    • Distance metrics (L2, cosine, dot product) determine how similarity is measured.
    • The curse of dimensionality makes traditional B+ Tree and Hash indexes ineffective.
    • ANN algorithms (IVF, HNSW, PQ) trade exactness for speed, with configurable recall.
    • Hybrid systems combine vector search with scalar filtering and keyword search for production use.
    • Vector databases are essential infrastructure for AI applications, particularly RAG with large language models.
